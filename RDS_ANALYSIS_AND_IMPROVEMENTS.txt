1. ARCHITECTURAL REVIEW & ISSUES FOUND
================================================================================

1.1 STRUCTURAL ISSUES
─────────────────────────────────────────────────────────────────────────────

ISSUE #1: Missing Alternative Flows in Critical Use Cases
  Location: UC-04.1 (View Profile), UC-04.2 (Update Profile), UC-10 (Check-out)
  Severity: HIGH
  Details:
    - UC-04.1: No handling for partial data retrieval scenarios
    - UC-04.2: No explicit "Cancel" alternative flow documented
    - UC-10: No "Early Check-out" or "Delay Check-out" scenarios
  Impact: User experience is unclear; system behavior not defined for common scenarios

ISSUE #2: Inconsistent Exception Handling & Error Linking
  Location: UC-04.3, UC-13.3
  Severity: HIGH
  Details:
    - Exceptions not explicitly linked to specific flow steps
    - Unclear which step triggers which exception
    Example: UC-13.3 has E1 "Unauthorized Access" but doesn't specify if this
             applies to all steps or specific ones
    - Generic error messages ("generic error message") violate security best practices
    - Inconsistent exception numbering (E1, E3, E7 are reused across use cases
      with different meanings)
  Impact: Developers cannot implement precise error handling; debugging difficult

ISSUE #3: Incomplete Non-Functional Requirements (NFRs)
  Location: All use cases
  Severity: MEDIUM-HIGH
  Details:
    - No Performance SLAs (e.g., "Profile retrieval < 2 seconds")
    - No Availability requirements (e.g., "99.5% uptime")
    - No Security requirements (e.g., "SSL/TLS encryption", "Password hashing algorithm")
    - No Scalability targets (e.g., "Support 10,000 concurrent users")
    - No Recovery Time Objective (RTO) / Recovery Point Objective (RPO)
  Impact: Development team cannot validate if system meets business expectations

ISSUE #4: Vague Business Rules & Poor Traceability
  Location: UC-04.2, UC-13.1, UC-13.2, UC-13.3
  Severity: MEDIUM
  Details:
    - UC-04.2 references BR1,BR2,BR3,BR4 but flow steps don't explicitly map to rules
    - BR1 "Registered Users Only" copied across multiple use cases with identical ID
      but potentially different context (e.g., profile vs booking)
    - UC-13.3 "Update Booking Status" has confusing nested steps (6.1, 6.2, 7.1, 7.2)
      but doesn't distinguish between alternative flows and main flow steps
    - No traceability matrix linking flow steps → business rules
  Impact: Inconsistent implementation; business logic scattered and hard to maintain

ISSUE #5: Poorly Defined Postconditions
  Location: UC-13.1 (View Booking List)
  Severity: MEDIUM
  Details:
    - UC-13.1 postcondition: "The system database is accessible"
      This is a PRECONDITION, not a postcondition!
    - Postcondition should state: "Booking list is displayed to Staff"
    - Missing postcondition about what happens if book list is empty
  Impact: Unclear what success looks like; possible implementation errors

ISSUE #6: Ambiguous Naming Convention Violations
  Location: UC-13.3, UC-04.2
  Severity: LOW-MEDIUM
  Details:
    - UC-13.3: "Update Booking Status" ✓ Good
    - UC-04.2: "Update Profile" ✓ Good
    - UC-04.1: "View Profile" ✓ Good
    - UC-10: "Check-out Room" - Should be "Perform Room Check-out" or
              "Complete Room Check-out" (more explicit object)
    - "Manage Booking" (UC-13) - Too generic; should be parent use case name only
  Impact: Minor; naming is mostly compliant with Verb+Object convention

ISSUE #7: Security & Privacy Concerns
  Location: UC-04.1, UC-04.2, UC-04.3, UC-13
  Severity: HIGH
  Details:
    - UC-04.1: Mentions "Sensitive attributes (password and role)" but doesn't
      specify HOW the system prevents exposure
    - UC-04.2: Doesn't mention audit logging of profile changes
    - UC-04.3: "New password must be securely stored" - says WHAT but not HOW
      (which algorithm? bcrypt, Argon2, etc.)
    - UC-13.3: No mention of audit trail for booking status changes
    - No rate limiting mentioned for failed login attempts (in exception E1)
    - No GDPR/data privacy considerations documented
  Impact: Incomplete security specifications; compliance risks; vulnerable implementation

ISSUE #8: Overlapping Business Rules with Unclear Context
  Location: UC-13.1, UC-13.2, UC-13.3
  Severity: MEDIUM
  Details:
    - UC-13.1 BR1 "Registered Users Only" - but Staff MUST be registered
      (precondition already states "authenticated as Staff")
    - Redundant precondition + business rule = confusion
    - UC-13.3 BR2 "Role-Based Access Control" vs precondition "user is authenticated
      as Staff" - what if a Staff member tries to update another Staff's booking?
  Impact: Circular logic; developers must infer actual intentions

ISSUE #9: Unclear System State Management
  Location: UC-04.2, UC-13.3
  Severity: MEDIUM
  Details:
    - UC-04.2: "The system refreshes the profile view" - what if refresh fails?
    - UC-13.3: Complex nested steps (6, 7, 9) suggest multiple interactions
      Should be separate use cases or clearer activity diagrams
    - Unclear if intermediate states are saved (e.g., after step 6.2, is booking
      in a temporary "Review Pending" state?)
  Impact: Race conditions possible; data inconsistency risks

ISSUE #10: Missing Preconditions for Logical Flow
  Location: UC-04.2, UC-13.3
  Severity: MEDIUM
  Details:
    - UC-04.2: Precondition "The system database is accessible" is implicit
      in all use cases; should it be at system level?
    - UC-13.3: Precondition "The booking is in a status that allows updating"
      is too vague - which statuses exactly?
  Impact: Implementation ambiguity; possible invalid state transitions

2.7 UC-13.3: Update Booking Status (MOST CRITICAL ISSUES)
─────────────────────────────────────────────────────────────────────────────

SEVERE STRUCTURAL PROBLEM:
  This use case is ACTUALLY MULTIPLE USE CASES:
  - UC-13.3.A: "Review Booking Details" (staff review → approve/reject)
  - UC-13.3.B: "Confirm Payment" (verify payment received)
  - UC-13.3.C: "Process Check-in Verification" (review evidence, assign room)
  
  These should be SEPARATE use cases in the system!

Flow Problems:
  - Step 6 (Review) leads to alternative steps 6.1, 6.2
  - Step 7 (Payment) leads to steps 7.1, 7.2
  - Then step 9 "System updates booking status" - when does system automatically update?
  
  This violates sequential flow logic. Suggested refactor:
    Main Flow: Steps 1-5 (access and select status update)
    Alternative Flow A1: "Approve Booking & Confirm Payment" (steps 6-9 scenario)
    Alternative Flow A2: "Reject Booking" (staff clicks "Reject")
    Alternative Flow A3: "Request Additional Information" (missing docs)

Missing Elements:
  ✗ Alternative flow: "Reject Booking" (with reason, notify customer)
  ✗ Alternative flow: "Request Additional Documentation" (ID verification fails)
  ✗ Alternative flow: "Escalate to Admin" (conflict or special approval needed)
  ✗ Exception: "Customer cancels booking during review" (real-time sync issue)
  ✗ Precondition: "Required documents uploaded by customer" (missing!)
  ✗ Precondition: "Clear definition of which statuses can transition to which"
  ✗ Audit trail: "All status changes logged with timestamp and staff ID"
  ✗ Notification: "Customer notified of status changes"
  ✗ Valid status transition: Specify the state machine
    Example: Pending → Confirmed → Payment Confirmed → Check-in Verified 
                                 → Checked-in → Checked-out

Status Transition State Machine (Missing):
  Need explicit diagram/table:
  ┌─────────────────────────────────────────────────────────────────┐
  │ From Status      │ To Status           │ Conditions            │
  ├─────────────────────────────────────────────────────────────────┤
  │ Pending          │ Confirmed           │ All docs reviewed     │
  │ Confirmed        │ Chờ thanh toán      │ Ready for payment     │
  │ Chờ thanh toán   │ Đã thanh toán       │ Payment confirmed     │

Clarity Issues:
  - Vietnamese status names mixed with English ("Chờ thanh toán", "Đã thanh toán")

  - Step 11 is incomplete: "Staff clicks 'Duyệt & Giao phòng'" - then what?

UC-04.1: VIEW CUSTOMER PROFILE
═══════════════════════════════════════════════════════════════════════════════

a. FUNCTIONAL DESCRIPTION
─────────────────────────────────────────────────────────────────────────────

UC ID & Name:       UC-04.1 View Customer Profile
Created By:         [Author]                    Date Created:  [Date]
Last Modified:      [Date]                      Version:       2.0

Primary Actor:      Customer
Secondary Actors:   None

Brief Description:
  The Customer accesses and views their personal profile information in a 
  secure, read-only presentation format. Profile data includes account 
  details, contact information, and verified identity information, but 
  excludes sensitive system attributes.

Trigger:
  - User selects "Tài khoản của tui" (My Account) from the account menu
  - User navigates directly via bookmark/URL after authentication

Preconditions:
  1. Customer is authenticated with valid session (session token not expired)
  2. Customer account exists and is in ACTIVE status (not suspended/deleted)
  3. Database connection is available and responding
  4. System has successfully validated Customer role (RBAC check passed)

Postconditions (Success Case):
  1. Read-only profile information is displayed to Customer
  2. No profile modifications were made
  3. Page load time < 2 seconds (NFR)
  4. Audit log entry created: "Customer [ID] accessed profile at [timestamp]"

Postconditions (Failure Case):
  1. Error message displayed (non-technical, customer-friendly)
  2. Customer remains on previous page or directed to dashboard
  3. System administrator notified if database error detected

c. ALTERNATIVE FLOWS
─────────────────────────────────────────────────────────────────────────────

ALTERNATIVE A1: Profile View with Session Timeout Warning
─────────────────────────────────────────────────────────────────────────────
Trigger: Session expiration within next 5 minutes detected
Occurs at: Step 2

A1.1  System detects session expiry in < 5 minutes
A1.2  Display warning banner: "Your session will expire in [X minutes]. 
      Click here to extend." (GRACEFUL DEGRADATION)
A1.3  Customer sees warning + profile data can be viewed
A1.4  If customer clicks "Extend": Reset session timer, clear warning
A1.5  If customer takes no action: Proceed to E1 (Unauthorized Access)

ALTERNATIVE A2: Profile Partially Available (Fallback Mode)
─────────────────────────────────────────────────────────────────────────────
Trigger: Partial database retrieval failure (some fields fail to load)
Occurs at: Step 3

A2.1  System retrieves core profile (name, email, phone) successfully
A2.2  Secondary fields (address, verification status) fail to load
A2.3  Display available data with notice: "Some profile information 
      unavailable. Please try again later."
A2.4  Provide "Retry" button to refresh secondary data
A2.5  Customer can view available profile data; no modification allowed
A2.6  Log technical error for investigation (ERROR level)

ALTERNATIVE A3: Offline Mode (Cached Session Data)
─────────────────────────────────────────────────────────────────────────────
Trigger: Database temporarily unavailable (E3 triggered) AND 
         previous page load successful
Occurs at: Step 3

A3.1  System attempts database query
A3.2  Request timeout after 3 seconds (NFR: max wait time)
A3.3  If cache available: Display cached profile data with 
      [CACHED - Last updated: [date]] notice
A3.4  Disable editing options (read-only maintained)
A3.5  Show: "Connection issue. Using last saved data. Refresh when online."
A3.6  When connection restored: Automatic refresh of page data


d. EXCEPTIONS & STEP LINKAGE
─────────────────────────────────────────────────────────────────────────────

EXCEPTION E1: Unauthorized Access (Invalid/Expired Session)
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 2 (session validation)
Conditions:
  - Session token expired (TTL exceeded)
  - Session token tampered (signature invalid)  
  - Session token revoked (user logged out from another device)
  - Concurrent session limit exceeded

System Behavior:
  E1.1  Log security event: "Unauthorized session access attempt from [IP]"
  E1.2  Invalidate current session immediately
  E1.3  Redirect Customer to Login page with message:
        "Your session has expired. Please log in again."
  E1.4  Preserve intended navigation: Next login redirects to profile page
  E1.5  If suspicious activity detected: Alert Admin
  
Recovery: Customer re-authenticates with valid credentials


EXCEPTION E2: Insufficient Permissions (Role Mismatch)
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 2 (RBAC validation)
Conditions:
  - User role is not "Customer" (e.g., Staff/Admin accessing as Customer)
  - Permission bits do not include "read_own_profile"

System Behavior:
  E2.1  Log authorization failure: "RBAC violation: [user_id] attempted 
        profile access with insufficient role"
  E2.2  Deny access - display: "You do not have permission to view profiles."
  E2.3  Redirect to user's appropriate dashboard (Staff → Staff Dashboard)
  E2.4  Alert Admin if repeated unauthorized attempts
  
Recovery: No automatic recovery; Admin intervention may be required


EXCEPTION E3: Database Retrieval Failure
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 3 (data retrieval)
Conditions:
  - Database connection timeout (no response after 3 seconds)
  - Database unavailable (500 error, connection refused)
  - Query execution failure (SQL exception)
  - Customer profile not found (deleted account scenario)

System Behavior (Non-Database Error):
  E3.1  Log error: "Profile retrieval failed - [error_type]: [details]"
  E3.2  Display customer-friendly message: 
        "We're having trouble loading your profile. Please try again in a moment."
  E3.3  Provide "Retry" button (client-side retry with exponential backoff)
  E3.4  After 3 failed retries: Show "Contact Support" link with ticket creation

System Behavior (Customer Profile Not Found):
  E3.5  If customer_id exists but profile deleted: Possible data corruption
  E3.6  Create urgent alert for database team
  E3.7  Display message: "Account data issue detected. Our team is investigating."
  E3.8  Offer contact support: "Email: support@booknow.com"

Recovery: System admin investigates database issue; customer may need account recovery

g. OTHER INFORMATION & DESIGN NOTES
─────────────────────────────────────────────────────────────────────────────

- Profile fields are displayed in read-only format at both UI and API levels
  (Defense in Depth principle)
- Payment method details (if stored): Show masked format only (****-****-****-4242)
- Identity verification status prominently displayed for compliance purposes
- "Last Updated" timestamp helps customer verify data accuracy
- No export functionality is provided (security: prevent data scraping)
- Profile changes made by other use cases (UC-04.2, UC-04.3) are NOT reflected 
  in real-time; customer must refresh page
- Session timeout warning (A1) improves UX by preventing sudden logout
- Fallback display of cached data (A2) increases resilience

UC ID & Name:       UC-04.2 Update Customer Profile
Created By:         [Author]                    Date Created:  [Date]
Last Modified:      [Date]                      Version:       2.0

Primary Actor:      Customer
Secondary Actors:   System (data validation engine), Database (persistence)

Brief Description:
  A logged-in Customer modifies their personal profile information to keep 
  account details current and accurate. The system validates all inputs 
  against data quality rules, prevents modification of sensitive fields 
  (email, role), and persists changes with audit trail.

Trigger:
  - Customer clicks "Chỉnh sửa profile" (Edit Profile) from profile view page
  - Customer navigates to profile settings section

Preconditions:
  1. Customer is authenticated with valid, unexpired session
  2. Customer account exists in ACTIVE status (not suspended)
  3. Customer account is not locked due to suspicious activity

  5. No other profile update in progress for this customer

d. EXCEPTIONS & STEP LINKAGE
─────────────────────────────────────────────────────────────────────────────

EXCEPTION E1: Unauthorized Access / Invalid Session
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 2 (session validation)
Conditions: Same as UC-04.1 E1

System Behavior:
  E1.1  Reject update request immediately
  E1.2  Clear form data (security: prevent cache leakage)
  E1.3  Redirect to Login page
  E1.4  Message: "Phiên làm việc hết hạn. Đăng nhập lại để tiếp tục."
  
Recovery: Re-authenticate; form reloaded from current database state


EXCEPTION E2: Concurrent Modification (Optimistic Lock Failure)
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 8 (version check)
Conditions:
  - Customer A loads profile (timestamp T1)
  - Customer A edits form (in browser, not aware of changes)
  - Customer B updates same profile (timestamp T2 > T1)
  - Customer A tries to save at T3
  - System detects last_modified at DB is T2, form has T1

System Behavior:
  E2.1  Reject update with specific message (not generic)
      "Another update occurred while you were editing. 
       Please review the current values and retry."
  E2.2  Display current database values alongside old values
  E2.3  Offer options:
        - "Refresh & Edit Again" (reload current values, clear form)
        - "Overwrite Changes" (force update, discarding other changes)
        - "Cancel" (discard my changes)
  E2.4  Log concurrency conflict for analysis
  
Recovery: Customer chooses action that matches their intent


EXCEPTION E4: Invalid Input Format or Empty Fields
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 6 (client validation) or Step 7 (server validation)
Conditions:
  - Empty required field (name, address)
  - Invalid format: Phone not matching pattern
  - Invalid format: Email already exists (if editable)
  - Text length: < 2 or > 200 characters
  - Special characters in fields that reject them (e.g., name)

System Behavior (Client-Side):
  E4.1  Display inline error message below problematic field
        Examples:
        - Phone: "Định dạng số điện thoại không hợp lệ 
                  (Sử dụng +84xxxxxxxxx hoặc 0xxxxxxxxx)"
        - Name: "Tên phải từ 2-50 ký tự"
        - Address: "Địa chỉ không hợp lệ (10-200 ký tự)"
  E4.2  Highlight field with red border
  E4.3  Disable Save button until all validations pass
  E4.4  Focus on first invalid field
  E4.5  No server request made (save bandwidth)

System Behavior (Server-Side):
  E4.6  If client validation bypassed: Reject request explicitly
  E4.7  Log security event: Potential tampering detected
  E4.8  Return 400 Bad Request with detailed validation errors (for debugging)
  E4.9  Do NOT make database update
  
Recovery: Customer corrects input values; can retry Save


EXCEPTION E5: Unauthorized Field Modification
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 7 (server validation)
Conditions:
  - Customer attempts to modify read-only field (email, role, account_status)
  - Detected by comparing allowed_fields list with submitted_fields

System Behavior:
  E5.1  Reject update with message: "Không thể thay đổi trường này"
        (You cannot modify this field)
  E5.2  Specify which field attempted modification:
        "Email address cannot be changed. It is your account login."
  E5.3  Log security event: Customer attempted unauthorized field change
  E5.4  If repeated attempts: Consider account lockdown
  E5.5  Preserve allowed field updates (partial success not appropriate here)
  
Recovery: Customer resubmits form with allowed fields only


EXCEPTION E6: File Upload Failure (Profile Photo)
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 7 (file validation)
Conditions:
  - File size exceeds 5MB limit
  - File type not in allowed list (JPG, PNG, WebP)
  - File contains malware signature
  - Storage quota exceeded for this customer
  - File upload connection interrupted

System Behavior:
  E6.1  Client-side pre-upload validation:
        - File size check before transmission
        - File type check (MIME type)
        - Message: "Kích thước ảnh không hợp lệ. Max 5MB. Chọn ảnh khác."
  E6.2  Server-side validation:
        - Verify file headers (magic bytes, not just extension)
        - Scan for malware (if available)
        - Verify file actually matches declared type
        - Check storage quota
  E6.3  If upload fails:
        - Offer "Retry Upload" option
        - Don't block other profile updates
        - Message: "Tải ảnh thất bại. Thông tin khác đã được lưu."
  E6.4  Delete partial/temporary uploaded files
  
Recovery: Customer retries photo upload separately


EXCEPTION E7: Database Update Failure
─────────────────────────────────────────────────────────────────────────────
Triggered at: Step 9 (transaction commit)
Conditions:
  - Database connection lost during update
  - Transaction timeout (update took > 30 seconds)
  - Constraint violation (e.g., duplicate phone if unique constraint exists)
  - Deadlock detected
  - Database in read-only mode (disaster recovery)

System Behavior:
  E7.1  Transaction automatically rolled back (no partial updates)
  E7.2  Customer-friendly message: 
        "Không thể lưu thay đổi. Vui lòng thử lại."
        (Cannot save changes. Please try again.)
  E7.3  Form data preserved for retry (not cleared)
  E7.4  After 3 failed attempts: Show "Contact Support" link
  E7.5  Log detailed error for database team investigation
  E7.6  Send alert to engineering: "Profile update failure rate: [n] errors"
  
Recovery: System admin resolves database issue; customer retries

g. OTHER INFORMATION & DESIGN NOTES
─────────────────────────────────────────────────────────────────────────────

- Email field is intentionally immutable because it serves as both:
  * System username/login identifier
  * Account recovery method (password reset emails sent here)
  Changing email would require complex side effects (update all recovery links, 
  notify of account recovery method change, etc.)

- Profile photo upload is optional; not required for profile save to succeed

- Concurrent modification exception (E2) provides user choice because:
  * Different scenarios require different solutions
  * User may want to see what changed and decide action
  * Automatic merge not safe for all fields

- Validation happens at TWO levels:
  * Client (responsive UX, save bandwidth)
  * Server (security, never trust client input)
  
  This defense-in-depth approach prevents:
  - Tampering (modified form values)
  - Injection attacks (SQL, XSS)
  - Bypassed validation (disabled JavaScript)

- Audit trail captures WHO changed WHAT and WHEN:
  * Compliance: Demonstrate data governance
  * Forensics: Investigate suspicious changes
  * UX: Show customer "Last updated" timestamp builds trust

g. OTHER INFORMATION & DESIGN NOTES
─────────────────────────────────────────────────────────────────────────────

Password Security Philosophy:
  - Passwords are the weakest link in authentication
  - Slow down the password change process deliberately:
    * Argon2 hashing takes ~500ms on purpose
    * Rate limiting prevents automated attacks
  - Defense in depth: Each step adds a layer
    * Client validation (UX)
    * Server validation (security)
    * Password history (account takeover prevention)
    * Account lockdown (brute force protection)

Rationale for Invalidating Other Sessions:
  - If customer changes password, all existing sessions should be revoked
  - Exception: Keep current session to maintain continuity
  - Prevents account takeover scenario:
    1. Attacker gains password
    2. Attacker logs in from new device
    3. Real owner changes password
    4. Attacker's session should be invalidated
  - Customer must re-authenticate on other devices → detects suspicious activity

Password Strength Feedback:
  - Don't use simple meter (weak/medium/strong) - too vague
  - Show exact requirements: "✓ Length >= 8 ✓ Uppercase ✗ Number"
  - Help customer understand what they need to add/change

Why Argon2 instead of bcrypt:
  - Argon2: Memory-hard, resistant to GPU/ASIC brute force (modern, 2015)
  - bcrypt: Slower but can still be brute-forced with specialized hardware
  - Scrypt: Middle ground, also acceptable
  - Password hashing is ~5-10% of security cost; worth using latest

Email Notification Timing:
  - Send non-blocking: Async task, don't wait for email to complete
  - If email fails: Log error but still show success to user
  - Customer will know if change succeeded from successful redirect
  - Email is notification, not confirmation

     - Grace period default: 60 minutes (configurable)
     - Late fee calculation: hourly_rate × hours_late
     - Example: $100/night = $6.25/hour late fee

BR5  Terminal State (NEW)
     - "Checked-out" status cannot be reversed by customer
     - Only admin intervention can reopen booking (for disputes)
     - Prevents accidental "un-checkout"

BR6  Atomic Operations (NEW)
     - All updates happen together or none at all (all-or-nothing)
     - Prevents inconsistent state (e.g., booking closed but room not marked clean)
     - Database transaction ensures consistency

UC-15: COMPLETE HOUSEKEEPING TASK
═══════════════════════════════════════════════════════════════════════════════

a. FUNCTIONAL DESCRIPTION
─────────────────────────────────────────────────────────────────────────────

UC ID & Name:       UC-15 Complete Housekeeping Task
Created By:         [Author]                    Date Created:  [Date]
Last Modified:      [Date]                      Version:       2.0

Primary Actor:      Staff (Housekeeping Worker)
Secondary Actors:   Admin (QA verification), System (room status update)

Brief Description:
  Housekeeping staff completes assigned cleaning tasks by following a checklist,
  documenting work with before/after photos, and submitting completion.
  Task status transitions: ASSIGNED → IN_PROGRESS → COMPLETED, and room 
  becomes available for new bookings.

Trigger:
  - Staff accesses "Nhiệm vụ của tôi" (My Tasks) from Staff Dashboard
  - App notification: "Task [Room 301] ready for completion"
  - Manager escalates: "Complete this task within 2 hours"

Preconditions:
  1. Staff member is authenticated with Employee role
  2. Task exists and is assigned to this staff member (status: ASSIGNED or IN_PROGRESS)
  3. Room is physically ready for cleaning (checkout completed, guests departed)
  4. Staff has access to room (physical key or digital lock)
  5. Device has camera for photo documentation

Postconditions (Success):
  1. Task status updated to COMPLETED
  2. Room status updated to READY (available for new bookings)
  3. Before/after photos stored in task record
  4. Checklist items marked as complete
  5. Time spent logged (started_at, completed_at)
  6. Task completion notified to Manager for QA review

ALTERNATIVE A1: Pause & Resume Task
─────────────────────────────────────────────────────────────────────────────
Trigger: Staff needs break or must move to higher priority task
Occurs at: Step 8-12 (during cleaning work)

A1.1  Staff clicks [Pause Task] button (within task screen)

A1.3  Task status remains "IN_PROGRESS" (not abandoned)

A1.6  System resumes timer (calculates total work time correctly)
A1.7  Staff continues from where left off

ALTERNATIVE A2: Report Issue / Request Help
─────────────────────────────────────────────────────────────────────────────
Trigger: Staff encounters damage, infestation, or cannot complete task
Occurs at: Step 8-12

A2.1  Staff clicks [Report Issue] button
A2.2  System displays issue report form:
      - Category: "Damage" / "Cleanliness Problem" / "Equipment Issue" / "Other"
      - Description: text + photo
      - Severity: "Low" / "Medium" / "High" / "Urgent"
A2.3  Staff fills form: "Found bed broken, cannot complete task"
A2.4  Staff submits issue report with photo evidence
A2.5  System:
      - Pauses task automatically
      - Notifies manager immediately
      - Creates maintenance work order if needed
      - Marks room as "Issue Reported" (unavailable)
A2.6  Manager responds with instructions:
      - "Call maintenance first, then resume cleaning"
      - OR "Mark room OUT_OF_SERVICE for repairs"
      - OR "Assign different staff to room"
A2.7  Staff resumes or escalates based on manager instruction

ALTERNATIVE A3: Incomplete Cleaning (Quality Assurance)
─────────────────────────────────────────────────────────────────────────────
Trigger: Manager reviews completed task & finds issues
Occurs at: After Step 18 (post-completion)

A3.1  Manager views completed task photos in dashboard
A3.2  Manager detects issue: "Bathroom mirror still streaky"
A3.3  Manager clicks [Request Rework] on task
A3.4  System:
      - Reverts task status: COMPLETED → IN_PROGRESS
      - Room status: READY → CLEANING_SCHEDULED (blocks new bookings)
      - Notifies original staff
A3.5  Staff receives notification: "Manager found issues. Please rework Room 301"
A3.6  Staff applies same flow (Steps 6-14) to fix issues
A3.7  Takes new photos, resubmits
A3.8  Manager approves or rejects again (max 3 cycles)

ALTERNATIVE A4: Task Reassignment Due to Absence
─────────────────────────────────────────────────────────────────────────────
Trigger: Assigned staff absent, task not started
Occurs at: Step 3 (staff logs in, task expected)

A4.1  Staff who should be assigned is absent
A4.2  Manager reassigns task to available staff (UC-14.A3)
A4.3  New staff logs in, sees task in "Reassigned to Me" section
A4.4  New staff can start task immediately
A4.5  Original staff_id preserved in audit log (for productivity tracking)
