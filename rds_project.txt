
1.4 UC-04: Manage Profile
1.4 UC-04.1 View Profile
a. Functional Description

UC ID and Name:	UC-04.1 View Profile
Created By:	Phùng Đình Khang	Date Created:	14/01/2026
Primary Actor:	Customer	Secondary Actors:	None
Brief Description::	This use case allows an authenticated Customer to view their personal profile information stored in the system, including basic personal details and account-related data, in a read-only mode.
Trigger:	The use case is triggered when the Customer selects the “Tài khoản của tui” option from the account menu while logged into the system.
Preconditions:	1. The user is authenticated as a Customer.
2. A valid Customer session exists.
3. The system database is accessible.
Postconditions:	1. The Customer’s profile information is displayed successfully in read-only mode.
Normal Flow	1. The Customer selects “Tài khoản của tui” from the account menu.
2. The system validates the active Customer session.
3. The system retrieves the Customer’s profile information from the database.
4. The system displays the profile information in read-only format.
Alternative Flows:	None
Exceptions:	E1 – Unauthorized Access
- The Customer session is invalid or expired.
- System behavior:
1.	The system redirects the user to the Login screen.
2.	A message is displayed indicating that authentication is required.
E3 – Data Retrieval Failure
- The system fails to retrieve profile information due to a database or system error.
- System behavior:
1.	The system displays a generic error message.
2.	No profile data is shown.
Priority:	Medium
Frequency of Use:	Medium
Business Rules:	BR1,BR2, BR3
Other Information:	Profile information is displayed in read-only mode. Sensitive attributes (such as password and role) are not visible to the Customer.


b. Business rules

ID	Business Rule	Business Rule Description
BR1	Registered Users Only	Only users with registered accounts can view profile information.
BR2	Role-Based Access Control	Customers must not view Admin or Staff profile information.
BR3	Only view the customer's own information.	Customers are not allowed to view other people's information.

1.4 UC-04.2 Update Profile
a. Functional Description

UC ID and Name:	UC-04.2 Update Profile
Created By:	Phùng Đình Khang	Date Created:	14/01/2026
Primary Actor:	Customer	Secondary Actors:	None
Brief Description::	This use case allows an authenticated Customer to update their personal profile information stored in the system, ensuring that personal data remains accurate and up to date.
Trigger:	The use case is triggered when the Customer selects the “Edit / Update Profile” option from the Profile screen while logged into the system.
Preconditions:	1. The user is authenticated as a Customer.
2. A valid Customer session exists.
3. The system database is accessible.
Postconditions:	Success:
1.	The updated profile information is successfully saved in the system.
2.	The system displays the updated profile details.
Normal Flow	1. The Customer navigates to the Profile screen.
2. The Customer selects “Edit Profile”.
3. The system displays the current profile information in editable fields.
4. The Customer updates one or more profile fields.
5. The Customer selects “Save Changes”.
6. The system validates the updated input data.
7. The system updates the Customer profile in the database.
8. The system displays a success message.
9. The system refreshes the profile view with the updated information.
Alternative Flows:	None
Exceptions:	E1 – Unauthorized Access
- The Customer session is invalid or expired.
- System behavior:
1.	The system redirects the user to the Login screen.
2.	A message is displayed indicating that authentication is required.
E4 – Invalid Input Format or Empty Fields
- One or more updated fields do not follow the required format (e.g., invalid phone number) or Empty Fields.
- System behavior:
1.	The system rejects the update request.
2.	Inline validation messages are displayed.
3.	The update process is not executed.
E7 – Database Update Failure
- The system fails to update the profile due to a database or system error.
- System behavior:
1.	The system displays a generic error message.
2.	Existing profile information remains unchanged.
Priority:	Medium
Frequency of Use:	Medium
Business Rules:	BR1,BR2,BR3,BR4
Other Information:	Certain sensitive fields such as email address, role, and account status may be read-only and cannot be modified by the Customer.
b. Business rules

ID	Business Rule	Business Rule Description
BR1	Registered Users Only	Only users with registered accounts can view profile information.
BR2	Role-Based Access Control	Customers must not modify Admin or Staff profile data.
BR3	Data Validation Enforcement	All updated profile data must comply with system-defined validation rules.
BR4	Email Immutability	The Customer is not allowed to update the email address because the email serves as the system username and unique account identifier.

1.4 UC-04.3 Change Password
a. Functional Description

UC ID and Name:	UC-04.3 Change Password
Created By:	Phùng Đình Khang	Date Created:	14/01/2026
Primary Actor:	Customer	Secondary Actors:	None
Brief Description::	This use case allows a logged-in Customer to change their current password by providing the existing password and setting a new one, ensuring continued secure access to the system.
Trigger:	The use case is triggered when the Customer selects the “Change Password” option from the Profile Settings screen.
Preconditions:	1. The Customer has a registered and active account in the system.
2. The Customer is currently logged in.
3. The system database is available.
Postconditions:	1. The Customer’s password is successfully updated.
2. The new password takes effect immediately for subsequent logins.
Normal Flow	1. The Customer selects “Change Password” from Profile Settings.
2. The system displays the Change Password screen.
3. The Customer enters the current password.
4. The Customer enters and confirms a new password.
5. The system validates the current password.
6. The system validates the new password against password rules.
7. The system updates the password in the User Database.
8. The system displays a success confirmation message.
Alternative Flows:	A4 – Cancel Change Password
1. The Customer selects “Cancel”
2. The system returns the Customer to the ProfileSettings screen.
Exceptions:	E3 – Incorrect Current Password
- The entered current password is incorrect.
- System behavior::
1.	The system displays the message “Current password is incorrect.”
2.	The user remains on the Customer to the Profile Settings screen.
E6 -   New Password Validation Failed
- The new password does not satisfy password rules.
- System behavior:
1.	The system displays a system error message.
2.	The user remains on the Customer to the Profile Settings screen.
Priority:	High
Frequency of Use:	Medium
Business Rules:	BR1,BR2
Other Information:	The new password must be securely stored using encryption or hashing mechanisms.

b. Business rules

ID	Business Rule	Business Rule Description
BR1	Valid New Password	 The Customer’s new password must be at least 8 characters long and contain at least one uppercase letter, one lowercase letter, and one number.
BR2	Current Password Verification	The Customer must provide the correct current password before changing to a new password.

1.10 UC-10 Check-out Room
a. Functional Description

UC ID and Name:	UC-10 Check-out Room
Created By:	Phùng Đình Khang	Date Created:	24/01/2026
Primary Actor:	Customer	Secondary Actors:	None
Brief Description::	This use case allows an authenticated Customer to check out from a booked room, indicating the end of their stay. The system updates the booking status, finalizes charges, and notifies relevant operational processes.
Trigger:	The use case is triggered when the Customer selects the “Check-out” option from the booking details screen or booking management interface.
Preconditions:	1. The user is authenticated as a Customer.
2. The Customer has an active booking with status “Checked-in”.
3. The current date and time are on or after the scheduled check-out time.
Postconditions:	Success:
1.	The booking status is updated to “Checked-out”.
2.	The stay is marked as completed.
3.	A checkout confirmation is displayed to the Customer.
Normal Flow	1. The Customer navigates to My Bookings.
2. The Customer selects an active booking with status “Checked-in”.
3. The Customer selects the “Check-out” option.
4. The system displays checkout details and confirmation information.
5. The Customer confirms the check-out request.
6. The system validates the booking status and check-out conditions.
7. The system updates the booking status to “Checked-out”.
8. The system displays a successful check-out confirmation.
Alternative Flows:	None
Exceptions:	E1 – Unauthorized Access
- The Customer session is invalid or expired.
- System behavior:
1.	The system redirects the user to the Login screen.
E7 – System Update Failure
- The system fails to update booking or room status due to a system or database error.
- System behavior:
1.	The system displays a generic error message.
2.	No state changes are applied.
Priority:	Medium
Frequency of Use:	Medium
Business Rules:	BR1,BR2,BR3
Other Information:	Check-out is a one-time action per booking. Once completed, the booking cannot be reverted to a previous state by the Customer.
b. Business rules

ID	Business Rule	Business Rule Description
BR1	Registered Users Only	Only users with registered accounts can view profile information.
BR2	Customer Role Authorization	Only Customers can check out their own bookings.
BR3	Valid Booking State	Check-out is only allowed for bookings with status “Checked-in”.

1.13 UC-13.3 Update Booking Status
a. Functional Description

UC ID and Name:	UC-13.3 Update Booking Status
Created By:	Phùng Đình Khang	Date Created:	24/01/2026
Primary Actor:	Staff	Secondary Actors:	Admin
Brief Description::	This use case allows a Staff member to update the status of a booking in the system in order to reflect the current stage of the booking lifecycle during daily operations.
Trigger:	The use case is triggered when the Staff selects the “Update Status” option from the Booking Detail screen.
Preconditions:	1. The user is authenticated as Staff.
2. The selected booking exists in the system.
3. The booking is in a status that allows updating.
4. The system database is accessible.
Postconditions:	1. The booking status is updated successfully.
2. The new status is reflected in the booking details.
Normal Flow	1. The Staff accesses the Booking Detail screen.
2. The Staff selects “Update Booking Status”.
3. The system displays the list of available booking statuses.
4. The Staff selects a new status.
5. The Staff confirms the status update.
6. Staff reviews booking details and submitted information.
    6.1. Staff clicks “Duyệt”
    6.2. System updates booking status to Chờ thanh toán
7. The system checks payment information associated with the booking.
    7.1. Staff clicks “Xác nhận thanh toán”
    7.2. System updates booking status to Đã thanh toán
8. The system records the successful payment confirmation.
9. System updates booking status to Xác thực check-in.
10. Staff reviews submitted check-in evidence
11. Staff clicks “Duyệt & Giao phòng”
Alternative Flows:	A4 – Cancel Status Update
1.	The Staff selects “Cancel”.
2.	The use case resumes at step 4 of the Normal Flow.
Exceptions:	E1 – Unauthorized Access
- The Staff session is invalid or expired.
- System behavior:
1.	The system redirects the user to the Login screen.
E1.1 – Booking Not Found
- The selected booking does not exist or is no longer available.
- System behavior:
  The system displays an error message and redirects the Staff to the Booking List screen.
E2 – Insufficient Permission
- The user does not have permission to update booking status.
- System behavior:
  The system denies the action and displays an authorization error message.
E3 – Booking Not Updatable
- The booking is in a final state that does not allow status changes.
- System behavior:
  The system disables the Update Status option and displays a warning message.
E4 – Invalid Status Transition
- The selected status violates predefined booking lifecycle transitions.
- System behavior:
  The system rejects the selection and prompts the Staff to choose a valid status.
E6 – Incomplete Booking Information
- Required booking information is missing or invalid.
- System behavior:
  The system prevents approval and highlights missing information.
E7 – System Update Failure
- The system fails to update the booking status due to a system or database error.
- System behavior:
1.	The system displays a generic error message.
2.	The booking status remains unchanged.
E8 – Payment Information Not Found
- No payment information is associated with the booking.
- System behavior:
  The system prevents payment confirmation and displays an error message.
E9 – Invalid Payment Status
- The payment is pending, failed, or invalid.
- System behavior:
  The system rejects the confirmation and keeps the booking status unchanged.
E10 – Invalid Status Order
- The booking has not been fully paid.
- System behavior:
  The system prevents the status update and displays a validation message.
Priority:	High
Frequency of Use:	High
Business Rules:	BR1,BR2,BR3
Other Information:	Booking status updates are logged for audit and tracking purposes. Status changes may trigger downstream operational processes .
b. Business rules

ID	Business Rule	Business Rule Description
BR1	Registered Users Only	Only authenticated users can access booking details.
BR2	Role-Based Access Control	Staff can update booking status but cannot perform Admin-level actions.
BR3	Valid Status Transition	Booking status must follow predefined lifecycle transitions
